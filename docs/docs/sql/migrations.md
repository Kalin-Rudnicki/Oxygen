# Migrations

Oxygen migrations are **filesystem-first**: your code only ever declares the *latest* schema. The
migration *history* lives on disk as one file per version, generated by **diffing your current
tables against the committed files**. This removes the biggest pain of code-defined migrations —
keeping every historical version of your models around in source.

Two responsibilities, cleanly split:

- **Generate / verify** (dev & CI) — diff code against the committed files; write a new file or fail
  if they've drifted. Database-free.
- **Apply** (runtime) — read the committed files and run any the database hasn't executed yet.

## Declare your schema

A `MigrationSchema` is just the current set of tables. `of` accepts table companions and/or
`TableRepr`s, mixed:

```scala
import oxygen.sql.migration.model.MigrationSchema

val schema: MigrationSchema =
  MigrationSchema.of(UserRow, ConnectionRow, PostRow, CommentRow)
```

## The guard spec

Extend `DbMigrationSpec` (from `oxygen-sql-test`) to manage a project's migrations as a test — the
"avro paradigm". It diffs `schema` against the files in `migrationPath` and:

- **passes** when they're up to date,
- **writes the new migration file** (and passes) when `OXYGEN_MIGRATION_ALLOW_UPDATE` is set,
- otherwise **fails the build** with the pending migration — so CI catches an uncommitted schema
  change.

```scala
import oxygen.sql.migration.model.MigrationSchema
import oxygen.sql.test.DbMigrationSpec

object MyDbMigrationSpec extends DbMigrationSpec {
  override def migrationPath: String = "modules/my-app/src/main/resources/migrations"
  override def schema: MigrationSchema = MigrationSchema.of(UserRow, PostRow)
}
```

This check is **database-free** (it compares code against the filesystem).

### Workflow

```bash
# CI / a teammate: just run the test — fails if migrations are stale
sbt "my-app/test"

# You changed a model class: regenerate, then commit the new file
OXYGEN_MIGRATION_ALLOW_UPDATE=true sbt "my-app/testOnly *MyDbMigrationSpec"
```

| Env var | Effect |
|---------|--------|
| `OXYGEN_MIGRATION_ALLOW_UPDATE` | write the new migration file instead of failing |
| `OXYGEN_MIGRATION_ALLOW_INCOMPATIBLE` | additionally allow a **breaking** change to be written |

## Versioning & compatibility

Versions are `oxygen.core.Version` (semver). The generator classifies each change and bumps
accordingly — the first migration is `1.0.0`:

| Change | Compatibility | Bump |
|--------|---------------|------|
| Create table / schema / extension; add **nullable** column; add non-unique index; drop index/FK | backwards-compatible | **minor** |
| Drop table / column / schema; add **non-nullable** column; tighten to `NOT NULL`; add unique index or FK to an **existing** table | incompatible | **major** |

A migration is *incompatible* if **any** of its changes is — and incompatible migrations require
`OXYGEN_MIGRATION_ALLOW_INCOMPATIBLE` to be written. Constraints (unique index, FK) on a table
created in the *same* migration stay compatible, since the table is empty.

> **No default values yet.** A non-nullable column add is always classified incompatible, because a
> `NOT NULL` column with no default breaks existing rows. Default-value support is planned.

## Applying at runtime

`MigrationService` reads the committed files and applies any the database hasn't run. Point it at
the directory via `MigrationConfig`, provided as a layer from your app config:

```scala
final case class MigrationConfig(atomicity: MigrationConfig.Atomicity, migrationsDir: String)
```

```scala
// from app config -> ZLayer
ZLayer.succeed(config.migration)   // MigrationConfig

// at startup
MigrationService.migrate                    // load dir, apply pending
MigrationService.migrate(MyApp.schema)      // ALSO verify the files reproduce the code first
```

`migrate(schema)` adds a **fail-fast** check: if the committed files don't reproduce your current
tables, startup fails with `MigrationsStale` rather than letting the app run against a stale schema.

### Atomicity

`MigrationConfig.Atomicity` controls transaction wrapping:

| Value | Behavior |
|-------|----------|
| `AllOrNothing` (default) | all pending migrations in one transaction |
| `PerMigration` | each migration in its own transaction |
| `None` | no transaction (not recommended) |

### Drift detection

Apply is strict about the filesystem and the database agreeing:

| Error | Meaning |
|-------|---------|
| `MissingMigration` | a migration recorded in the DB is absent from the filesystem |
| `MigrationsDiffer` | a committed file disagrees with what the DB recorded for that version |
| `OutOfOrderMigration` | a new file sorts *before* an already-applied version (not supported in MVP) |

## Local-dev "collapse"

Because generation diffs against whatever files are present, you can collapse history locally by
deleting files and regenerating: delete everything → one genesis migration; delete only the
unreleased tail → those changes fold into a single migration.

> **Collapse is local/greenfield only.** Deleting files that a real database has already executed
> will trip the drift checks above. Reset the database alongside the files; never collapse against a
> database you can't reset.

## File format

Each `<version>.json` is a self-contained `PersistedMigrationFile`: the full resulting schema
**snapshot** plus the **diff** that produced it (the snapshot is what makes future reverse/range
diffing possible). The `sql` is *not* stored — it's derived from the diff at apply time. You
normally never hand-edit these; let the guard spec regenerate them.
